6. The demo flow (this is the only path that must work end-to-end)

Exact steps from BRIEF.html Section 15. Reproduced here for convenience:

  1. User clicks Sarah 1:1. Brief renders in the main canvas — the room + insight A + two inline actions.
  2. User clicks "Show what shifted". Inline expand reveals a 3-line summary citing two source meetings, with an "Open full note →" link.
  3. User clicks "Bring this up". Button state changes; quiet inline confirmation: "Saved to your Sarah brief."
  4. User clicks a small "Start recording (demo)" affordance. The recording notification appears in the corner: "Recording started — bring up: Q3 launch assumptions". Match the visual style of 04_recording_notification.png exactly; change only the copy.
  5. The main canvas transitions from the Brief to the meeting note. The pinned bullet sits at the top: "Bring up: Q3 launch assumptions (two have shifted this week)", with the lightest possible marker indicating from your brief.
  6. User clicks back to the sidebar. Clicks Pricing review. Brief renders with insight B. No further action needed — this demonstrates breadth.
  7. User clicks Roadmap sync. Brief renders with insight E. Same — breadth demonstration only.

Interaction notes

  • Transitions: fade or subtle slide, ~150ms. Nothing hard-cut. Nothing showy.
  • The "Start recording (demo)" button is a prototype-only affordance — in real Granola it would auto-trigger from Zoom detection. Style it minimally; don't make it look like part of the product.
  • The recording notification auto-dismisses after ~4 seconds, but the pinned bullet persists.
  • State management: a single React state machine for which meeting is selected + current beat. Keep it tiny; this is a demo.

7. Visual fidelity rules (override everything else)

  1. Color, type, spacing, radius come from ./references/, not from training data and not from the existing dashcam prototype tokens (which are approximations). If they disagree, screenshots win and you should update src/index.css.
  2. Density: Granola is spacious. If a surface feels dense, remove something.
  3. The Brief should feel like a kind of note, not a separate UI primitive.
  4. No badges. No status pills with bright fills. No colored indicators of any kind.
  5. Shadows minimal. Hover states barely visible.
  6. No data viz, no graphs, no charts, no scores, no confidence percentages.
  7. The recording notification (Beat 2) must be visually indistinguishable from today's Granola recording notification except for the appended copy.
  8. The pinned bullet (Beat 3) should read as a Granola note bullet with the lightest possible marker — a small inline label or a subtle accent in the left margin. Nothing more.

8. Voice & tone rules

Every UI string must pass: would a human assistant say this?

  • Use the exact copy from BRIEF.html sections 5 and 15 for the insight strings. Don't paraphrase.
  • Inline confirmations are short and human: "Saved to your Sarah brief." Not "Item added successfully."
  • The "from your brief" marker is sentence-case lowercase: from your brief. Not FROM YOUR BRIEF, not Brief Item.
  • Button labels are verbs at the user level: "Bring this up", "Show what shifted", "Ping Pat for a quick read". Never "Submit", "Apply", "Confirm".

9. Out of scope (do not build, even if easy)

  • Anything in BRIEF.html Section 16.
  • Categories C and D from the insight portfolio (documented for context, not demoed).
  • A sidebar tab for any new view.
  • A chat panel.
  • Real Zoom/Meet detection (the "Start recording (demo)" affordance is enough).
  • Settings, account, auth, admin.
  • More than the three demo meetings working.
  • Mobile / responsive below 1024px.

10. Acceptance criteria

  • Running npm run dev and navigating to /?demo=brief loads the new flow without breaking the existing dashcam prototype.
  • All seven steps of Section 6 work end-to-end in a single uninterrupted demo run.
  • Visual fidelity passes the "is this Granola?" test against the screenshots in ./references/.
  • The three-beat arc (Brief → notification → pinned bullet) carries the same insight content through all three surfaces.
  • All three insight categories (A, B, E) appear, with their exact copy from BRIEF.html.
  • Every UI string passes the voice test in Section 8.
  • Zero new pieces of chrome added to the app outside the demo route.
